Trialogue v3 多智能体转录代理 — 7 层纵深防御架构

版本 v3 · 2026-03-24 · 已验证: Claude Code 2.1.81, Codex CLI 0.116.0

1. 它是什么

Trialogue 是一个多智能体转录代理（transcript broker）。两个来自不同厂商的 CLI 智能体通过转录协议进行协作；Broker 坐在中间，负责收集、清洗和路由转录内容。

核心设计原则：智能体之间绝不共享执行环境。协作通过制品交换完成，而不是文件系统访问。

威胁模型：系统假定任何智能体都可能被攻陷或产生对抗性输出。Broker 将所有智能体生成的内容视为不可信输入。

2. 七层纵深防御

防御层彼此独立——每一层的失败不会级联到下一层。

L1
转录清洗器 — 在提示词注入到达之前剥离结构性注入模式。5 条规则按序执行：不可见 Unicode 剥离、块包装器剥离（6 类注册标签）、审计头剥离、LLM 格式模式剥离（7 条锚定正则）、空白压缩。三种模式：strict（默认）/ permissive / disabled。
L2
沙箱隔离 — 每个智能体独立的 OS 用户、HOME、工作区、TMPDIR。Level 2 部署支持 sudo -u 权限降级。
L3
操作分类器 + 互斥锁 — Broker 管理的宿主共享资源互斥。操作分类器识别敏感操作并取锁。
L4
版本门 — 启动时 + 运行时二进制 SHA-256 校验。当前模式：warn（识别不在白名单的二进制并记录告警；升级到 enforce 是运维决策）。
L5
本地审计锚 — 仅追加（append-only）SHA-256 哈希链 + HMAC 签名。独立链验证工具：verify-summary-chain.py。
L6
远程审计锚 — 独立进程的 Sink HTTP 服务器（SQLite WAL），凭证隔离的 publish/verify 令牌。智能体不持有任何令牌。
L7
交叉审计 — 每轮对话摘要在智能体之间进行比对。

设计逻辑：L1 拦截最低成本的结构性注入。绕过 L1 的语义攻击会命中 L2（沙箱限制爆炸半径）、L3（锁约束宿主操作）、L5-L7（完整审计轨迹使所有行为可追溯）。

4. 关键数据流

4.1 正常对话轮次

  1. 用户在 chat.py 输入话题
  2. server.py 将消息分发给当前轮次的目标智能体
  3. launcher.sh（Claude）或 codex_app_server_runner.py（Codex）启动子进程
  4. 运行前执行版本门检查（L4）：比对二进制 SHA-256 和版本字符串
  5. 智能体输出经 L1 清洗器处理后路由到对方
  6. 每轮摘要写入本地哈希链（L5），异步发布到远程 Sink（L6）
  7. 定期执行交叉审计（L7）

4.2 审批流（Codex 0.116.0+）

Codex app-server 的审批请求分为两代协议：

协议 方法 决策枚举 处理方式
旧协议 item/commandExecution/requestApproval
item/fileChange/requestApproval
item/permissions/requestApproval		 accept / acceptForSession / decline / cancel 进入 Broker 审批队列，等待人工/策略决策
0.116.0+ 新协议 applyPatchApproval
execCommandApproval		 approved / approved_for_session / denied / abort 自动回复 {"decision": "denied"}（安全默认值）
协议漂移风险：两代协议使用不同的决策枚举。旧协议的 decline 不等于新协议的 denied； 旧协议的 accept 不等于新协议的 approved。混用会导致协议错误。 fake server 测试桩已对齐两套枚举，使用显式集合匹配而非前缀匹配。

4.3 远程审计锚

Broker (publish.token, 只写) │ │ POST /append ▼ Sink (独立进程, 独立用户) │ SQLite WAL, synchronous=FULL │ 序列号: 每房间无间隙递增 │ 幂等: 相同 (room_id, rid, sha) → 200 │ 冲突: 相同 (room_id, rid), 不同 sha → 409 │ │ GET /verify ▲ Broker (verify.token, 只读)

状态机：healthy → degraded（积压待发）→ anchor_blocked（硬上限超限/blocking 模式 3 次未锚定/verify 不一致）→ read_only_diagnostic（启动验证不一致）。恢复需要系统自愈 + 运维手动调用 reset_recovery()。

5. 版本门（L4）

版本门通过 runner-version-allowlist.json 配置：

{
  "policy": "warn",
  "runners": {
    "claude": {
      "versions": ["2.1.81 (Claude Code)"],
      "hashes": ["047e3f5..."]
    },
    "codex": {
      "versions": ["codex-cli 0.116.0"],
      "hashes": ["baefc10..."]
    }
  }
}

匹配逻辑（hardening.py:_version_rule_match()）：

  • 版本字符串：精确匹配 $BIN --version 的完整输出（不是裸 semver）
  • SHA-256 哈希：精确匹配二进制文件摘要
  • 任一匹配即通过：version in allowed_versions or sha in allowed_hashes
当前状态：warn 模式。不在白名单的二进制会被识别并记录告警，但不会被阻断。 这恢复了版本门的识别能力（此前白名单为空，等同透传），但强制阻断能力需要运维将策略升级到 enforce。

6. 清洗器详解

_sanitize_text_once() 按序执行 5 条规则：

#规则说明
1不可见 Unicode 剥离32+ 字符类（零宽、双向覆盖、变体选择器等）
2块包装器剥离基于栈的标签匹配器，6 类注册包装器。处理嵌套、交叉嵌套、孤立标签。不区分大小写。已测试 500 层嵌套，无 ReDoS。
3审计头剥离行级正则，匹配 [TRIALOGUE-AUDIT] 头
4LLM 格式模式剥离7 条锚定正则（ChatML、Llama 2、Copilot、Markdown 标题等）。块级匹配（^...$ + re.MULTILINE），避免行内误判。
5空白压缩3+ 连续换行 → 2

模式配置从 sanitizer-patterns.json 加载（支持热重载），代码级默认值作为兜底。

已知局限 (W11)

^###\s*System:\s*.*$ 模式会匹配 Markdown 代码块内的内容（re.MULTILINE 下 ^ 锚定到任意行首，不仅是块首）。 基准测试确认 1/10 LLM 格式讨论良性样本受影响。良性通过率：96.7%（在门限内）。这是有文档记录的取舍，不是静默故障。

11. 快速开始

# 复制配置
cp trialogue-v3.conf.example trialogue-v3.conf

# 启动 Broker + 智能体
bash start.sh "讨论主题"

# 启动 Web UI
bash start-web.sh

# 启动远程审计 Sink（独立进程）
bash start-remote-anchor-sink.sh

# 运行对抗测试
python3 hardening_adversarial.py
python3 hardening_adversarial_r2.py
python3 hardening_injection_benchmark.py
python3 hardening_p1_adversarial.py
python3 hardening_p1_recheck_adversarial.py
python3 hardening_protocol_smoke.py

# 独立验证哈希链完整性
python3 verify-summary-chain.py /path/to/room.jsonl
