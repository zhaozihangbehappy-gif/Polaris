# Codex 对 Shadow 的批判性、建设性意见

Codex 的核心判断是：**Shadow 当前最危险的地方，不是技术点太难，而是概念边界正在失焦。**

它认为，如果继续把 Shadow 同时当成“助手”“系统壳”“工作流引擎”“人格容器”“自动化入口”这类复合概念来推进，项目在概念上会越来越性感，但在工程上会越来越不可收敛。对 Codex 来说，当前最重要的不是继续扩大能力面，而是先砍定义：Shadow 到底优先解决哪个单点痛苦？是持续在线的执行代理、带记忆的个人操作系统入口，还是多通道消息协作体？这三者的架构、评估标准和失败模式完全不同，不能在同一阶段混着推进。

Codex 特别警告了一个常见错误假设：**默认把模型能力当作稳定基座。**

它认为，很多 brainstorm 天然会乐观假设：模型会持续理解上下文、会自己补齐缺失信息、会在长链路任务里保持一致意图、会自动在群聊/私聊/自动执行之间切换人格与权限。但现实里，这恰恰是最不稳定的部分。一个可靠系统不能建立在“模型大概懂”的前提上，而要建立在：

- 显式状态
- 可验证计划
- 明确权限边界
- 可回放日志
- 可审计执行轨迹

否则 Shadow 会很快变成一个 demo 很强、真实使用很不稳定的系统。

Codex 的第二个重点，是**缺失硬约束**。它认为今天这类项目最常见的失败，并不是能力做不到，而是没有提前写清楚“不允许发生什么”。比如：

- 什么情况下 Shadow 可以自主行动，什么情况下必须停下来问
- 哪些记忆能长期保存，哪些只能短期缓存
- 群聊里它是参与者、代理人，还是只做审计
- 外部动作的确认门槛是什么
- 一次任务允许消耗多少 token、多少时间、多少失败重试
- 出现歧义时，优先保守还是优先推进

没有这些硬约束，后续所有体验优化都会互相打架。

Codex 还专门批评了对“记忆”的直觉性理解。它认为，Shadow 需要的不是越来越多的上下文堆积，而是**分层记忆机制**：身份与偏好、任务态、项目态、短期现场态必须被拆开，否则迟早会出现污染、误召回和泄漏。如果未来要跨会话、跨渠道、跨项目工作，记忆系统至少要先回答三个问题：**谁能读、何时写、多久失效。** 没有记忆治理，系统不会越来越聪明，只会越来越自信地犯旧错。

在推进建议上，Codex 明确主张不要继续进行泛化讨论，而是进入 **约束驱动的收敛阶段**。它建议先写一份很短的 `Shadow v0` 定义文档，只包含：

- 目标用户
- 一个核心场景
- 三个非目标
- 五条硬约束
- 两类禁止行为

然后基于这份定义做一个最小闭环：输入任务、生成计划、执行有限工具、产出结果、留下可审计轨迹。评估也不要看“像不像人”，而要看：

- 成功率
- 可预测性
- 回滚成本
- 误动作率

并且从第一天开始积累失败样本库，把每次出错归因到提示、状态、工具、权限还是产品定义，而不是统统归因为“模型还不够强”。

Codex 的结论可以压成一句：

> 不要急着把 Shadow 做成“一个会成长的人”，先把它做成“一个边界清楚、行为可审计、在单一高价值场景里稳定工作的系统”。
